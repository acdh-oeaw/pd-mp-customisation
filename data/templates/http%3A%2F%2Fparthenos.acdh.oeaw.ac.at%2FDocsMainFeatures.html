Main Features

This page gives you an overview of the main features of PARTHENOS Discovery and how to use it.

The application offer multiple options to explore the dataset:

  • Keyword search with facets
  • Search based on semantic relations between entities (Query builder)
  • Explore by sources - Overview of the dataset divided by the provenance of the records
  • Explore via the underlying model
  • SPARQL - standard query language for RDF data, full expressive power at the cost of having to learn a query language
  • Custom detail views allows for lateral navigation, jumping between related entities

Options to search and explore

Keyword search with facets

This is the basic "google-like" search method, returning records matching the search term. Some notes of the usage:

  • By default the search matches on full tokens not parts of a word. To allow for partial match you need to suffix the search term with asterisk. Search query:

    corp*

    will match corpus but also corpora and any other words starting with corp.

  • If searching with multiple terms, only items are returned that match all of the terms. Example:

    corpus health

The potentially large result set may be further narrowed down through the use of so-called facets, a feature well-known from online shopping catalogues. Facets divide the result set along selected dimensions or fields, e.g. by type, by the source, or by the spatial coverage of the items in the result set. The available values of one facet including the number of matching items are listed in a sidebar on the left. By clicking on one of these values the result set will be filtered to items matching given value.

Search based on relations between entities (Query builder)

The query builder is a customizable UI component featured by Metaphactory supporting the users to formulate more complex queries based on relations between entities in an user-friendly, intuitive manner. The customization is needed to adapt the component to the specifics of ther underlying data model, but it mainly allows to abstract from internal intricacies and detail of the data model, by exposing only the most pertinent entities (in case of PARTHENOS Entities Model: Actor, Dataset, Software, Service, Place) and relations between these (e.g. Dataset createdBy Actor or Actor memberOf Actor/Group). Conceptually, this approach is based on the idea of Fundamental Categories and Relationships for querying CIDOC CRM based repositories, introduced by Tzompanaki and Doerr (2012)[1]. Metaphactory implements this concept consistently throughout the application, by introducing "Fields", further described in a separate section. More details on the configuration and functionality of the Query builder can be found on the help page of Metaphacts SemanticSearch.

Screenshot illustrating the functioning of the query builder: Search for actors who curated a specific dataset (in this case the Parthenos Policy Wizard Dataset)

Explore the data based on the provenance (source)

This is a special view of the dataset, forming a matrix, dividing the dataset along two dimensions: source of the records, main type of the entity, informing about how many entities of given type are being contributed by each of the data providers. The numbers in the table lead to a corresponding SPARQL query, listing entities of given type from given source.

Explore the data via the underlying model

Specific to an RDF-based knowledge graph, is that the model underlying the data (the ontology) is represented in the same form as the data itself. Thus a corresponding database (called "triplestore") and any applications operating on RDF-graphs can handle both the data and the model. This means that the entities of the model (called "classes" and "properties" or "predicates") can be explored in much the same way as the instance data, i.e. the individual described datasets, actors, services etc.

In PARTHENOS Discovery we took advantage of this feature providing views for the classes and properties of the model that feature the definition of given entity, its relation to other entities of the model (e.g. subclasses and superclasses) as well as its usage, i.e. showing sample entities of given class.

For example, you can explore the class Actor and in the tab "Instances" you see listed instances or individuals of type Actor. The tab "Usage" summarizes the relations or properties instances of Actor are described with. So for example the following table row:

has contact point Address 32582

This means that in 32582 cases Place instances have a property “has contact point” with an Address. When you click on the number you get the list of the corresponding Place-Address pairs.

SPARQL

SPARQL is the language for querying RDF data (specification). It requires some practice to master, but allows its user to query the data for any custom complex patterns. In Metaphactory, all the data displayed to the user is retrieved from the underlying triplestore via predefined SPARQL queries.
In many places in the user interface, the underlying SPARQL query is linked, allowing the user to run the query separately (in a dedicated generic SPARQL-query interface) and to adjust it further on demand. One very simple example of this is the option to adjust the size of the result set (with the keyword LIMIT):
When listing items in the user interface, the underlying queries are mostly restricted to 100 up to 1000 items for practical and performance reasons. When the user wants to see more items, this can be done by running the SPARQL query separately with the LIMIT adjusted accordingly.

Detail Views

While various means for exploration of and search in the dataset are necessary to find your way through the data space, in the end the most relevant information is the detailed description of individual resources. Detail Views allow you to inspect the available information about individual entities in the dataset, roughly corresponding to a metadata record or a landing page of a resource.

Next to a generic view (tab "Source statements") which lists all the properties of a given entity, for the main types custom views are available, which take into account the specifics of given type. For example, for Places there is a separate tab with a map, where the places are located on the geographical map (provided the corresponding spatial coordinates have been provided).
If there is an image representation for a given entity, it is also displayed in the header as a thumbnail.
In the tab "minimal metadata", most relevant properties of given entity, customized per type are displayed for quick overview.

Templates & Fields

Metaphactory allows the presentation of data in manifold ways, based on a powerful templating mechanism. The logic which template is used for rendering of which entity is governed by the type (Class) of the corresponding entity, respecting also the class hierarchy. This means that a template for an instance of a given class is applied to all instances of all subclasses as long as there is no more specific template for one of the subclasses.

More information on the templating mechanism of Metaphactory

The templating mechanism is accompanied by so-called Fields. Fields in Metaphactory are a means to hide the internal complexity of the underlying data model behind structures known and understandable by the normal user. Essentially this is an implementation of the concept of Fundamental Categories and Fundamental Relations [1]. In practice this means, the fields offer a transparent abstraction layer over the underlying data.

More information on the definition of fields in Metaphactory help page.

[1] Tzompanaki K. and Doerr M. (2012) Fundamental Categories and Relationships for querying CIDOC CRM based repositories. Technical Report ICS-FORTH/TR-429, April 2012. http://www.cidoc-crm.org/sites/default/files/TechnicalReport429_April2012.pdf
